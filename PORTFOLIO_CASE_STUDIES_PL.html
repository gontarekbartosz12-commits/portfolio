1 · Boty tradingowe Polymarket

Python · Polymarket Gamma & CLOB API · Polygon/Web3 (py-clob-client) · Simmer API · Linux VPS + cron · Telegram

Problem

Rynki predykcyjne źle wyceniają szybko zmieniające się wydarzenia. Chciałem sprawdzić — realnym kodem na realnych danych — czy konto detaliczne jest w stanie wypracować przewagę i które strategie przetrwają zderzenie z rynkiem.

Co zbudowałem

  • Copy-trader — śledzi czołowe portfele on-chain, wykrywa nowe pozycje przez data API Polymarket i kopiuje je jako % salda, z limitami ekspozycji.
  • Skaner dywergencji AI — handluje tylko, gdy niezależna wycena AI rozjeżdża się z ceną rynkową o ≥10 pp.
  • Model wyświetleń BeastBot — model ilościowy dla rynków YouTube (projekt 2).
  • Bot LP Limitless — automatyczne dostarczanie płynności na łańcuchu Base z własnym backtesterem i monitorem.

Inżynieria

  • Diffowanie pozycji — robi snapshot stanu portfela i reaguje tylko na deltę (nowa pozycja), nigdy nie kopiuje starych.
  • Warstwa ryzyka — % kopiowania, min/max rozmiar transakcji, limity dzienne i obsunięcia egzekwowane przed każdym zleceniem.
  • Odporność — każde wywołanie sieciowe ma timeouty + try/except; pętla sama się leczy, więc niestabilne API nigdy jej nie zabija.
Uczciwy wynik: najpierw tryb paper/$SIM. Najcenniejszy efekt był negatywny — audytując ~3 380 transakcji on-chain udowodniłem, że naiwny copy-trading jest na skalę detaliczną nieopłacalny (≈ −222 $/tydz. przy 300 $). To zabiło złą strategię zanim zaryzykowałem realny kapitał i skierowało projekt na przewagi oparte na modelach.

Pokazuje: realne integracje API + Web3, zarządzanie ryzykiem i dyscyplinę falsyfikowania własnych pomysłów danymi.

2 · BeastBot — model ilościowy rynków wyświetleń YouTube

Python 3.10+ · SciPy (dopasowanie sigmoidy) · YouTube Data API v3 · Polymarket Gamma · wycena rozkładem normalnym · konsensus wielu agentów

Przewaga

Wczesne dane o wyświetleniach (T+6h…T+24h) układają się w krzywą sigmoidalną. Dopasowuję ją, projektuję wysycenie wyświetleń, zamieniam na prawdopodobieństwo bucketu rozkładem normalnym i handluję tylko, gdy rynek rozjeżdża się z modelem o ≥8 pp.

sources/   → wyświetlenia na żywo (YouTube API) + scraper zapasowy
model/     → growth_curves (sigmoida) · baseline historyczny · bootstrap CI
hermes/    → 3 agentów (api / krzywa / historia) → ważony konsensus
pricer.py  → rozkład normalny → prawdopodobieństwa bucketów
trader.py  → główna pętla   ·   monitor.py → logika wyjścia
  • Skalibrowana niepewność — sigma rośnie dla dłuższych horyzontów (na bazie literatury o wzroście wyświetleń).
  • Konsensus — Hermes miesza trzy sygnały z jawnymi wagami; pomija rynki o niskiej pewności.
  • Czyste sekrety — każdy klucz przez os.getenv(), nic na sztywno.

Pokazuje: stosowaną statystykę/ML, czystą architekturę modułową i przejście od pomysłu badawczego do wdrażalnego, ograniczonego ryzykiem systemu.

10 · Tradebot — backtest swing-momentum

Python · vectorbt · pandas · yfinance · Jupyter

Rygorystyczny backtest zbudowany, by zdecydować, czy w ogóle warto budować bota produkcyjnego — z jawnymi bramkami, które powiedziały „jeszcze nie". I to zdyscyplinowane „nie" jest tu sednem.

  • Metoda — momentum kompozytowe + filtr ADX/SMA, wejścia bez look-ahead, trailing stop ATR, 5 bps poślizgu, walidacja walk-forward (6 foldów).
  • Bramki decyzyjne — Sharpe ≥ 0,5, max obsunięcie ≤ 15%, walk-forward Sharpe ≥ 0,3, ≥ 150 transakcji — fail dowolnej → nie wdrażaj.
  • Uczciwość — otwarcie dokumentuje biasy survivorship i look-ahead oraz ich mitygacje.

Pokazuje: quant-rygor i decyzje oparte na dowodach — dojrzałość, by nie wdrożyć stratnej strategii.
